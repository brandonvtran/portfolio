Quarter Harmony

A high fidelity prototype of a music management application with a focus on redesigning the playlist functionality. The process of creating the prototype included interviewing music application users, analyzing the pros and cons of existing music applications, and designing the music application at various fidelity levels.

Initially, an interview process took place in order to find the pros and cons of various applications and to find what users wished were in a music application. Next, an in depth analysis of different application's features and work flows occured in order to find what components of the application to keep and what needed improvement. Afterwards, low fidelity drawing prototypes were designed in order to plan what the workflow of our application should look like as well as what features we wanted to include.

Upon finishing the low fidelity prototypes, we began the skeleton of the high fidelity prototype. The skeleton revealed missing components of the application that were unseen in the drawings -- this included navigation indicators and task completion indicators.

The design process was next. Our color scheme was inspired by Apple Music's ombre header and our homepage's random playlist finder feature was inspired by Shazam. Hard design decisions took place such as the homepage's central feature -- we decided on the random playlist finder in order to accomodate the needs of people that we interviewed. Overall, the application focuses on collaboration amongst individuals to make the playlist making process more community oriented.

Ripple

An android and web application that streamlines a music queuing process. The "DJ" receives song requests, on his or her android application, from the "collaborator" who makes the song requests on his or her mobile web application. It is then the DJ's responsibility to either accept or decline the request such that the song will either be added or not added to the queue to be played. Acted as the Senior Systems Analyst and was responsible for product documentation and planning.

The project began by brainstorming ideas to serve the purpose of simplifying the lives of students. Once the idea of a music queueing application was conceived, use cases and user stories were documented to plan the components necessary for the application.

A diagram was designed to illustrate the possible workflows of the application.

After the initial planning of the application, was the implementation of the application. Ripple utilizes the of Java, XML, Javascript, HTML, and CSS as well as a Firebase database. The process of building the application was done through the software engineering methodology: agile.

In order to ensure the application worked without defects, acceptance tests were performed.

ReMemo

A Javascript, HTML, CSS based mobile and web application designed to help people remember their tasks for the day and improve their memory

As a group, we brainstormed for an idea for a project that would serve the purpose of helping folks above the age of forty. We came up with an idea for a mobile and web application that helped the user remember tasks for the day and improve memory for the long term. In order to elaborate on the application's purpose in the real world, and plan how it worked, we created a storyboard.

After planning the project, we created a prototype. After testing the protype, we implemented the functionality of the web application by utilizing Javascript, CSS, and HTML. The application was designed to be easy to be simple yet functional. When the application was finished, we performed usability tests on peers. Upon receiving data on design flaws, we redesigned specific aspects of the application.

To ensure that the application works on the target users, we tested ReMemo on users who claimed that they had poor memory. A protocol was designed to ask the user to do various tasks. By performing usability tests, we were able to not only find bugs within the application, but we were also able to find aspects of our application that were confusing to the user. We made the appropriate fixes upon finding the flaws in ReMemo.
